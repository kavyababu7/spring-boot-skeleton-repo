# AI Coding Guidelines for this Repository

When writing code in this repository, always adhere to the following architectural standards and best practices:

## General Principles
- **Package Structure**: Always follow the established package structure under `com.nissan.template`.
- **Java Version**: Use Java 21 features where appropriate.
- **Lombok**: Use Lombok annotations (`@Data`, `@Getter`, `@Setter`, `@RequiredArgsConstructor`, `@Slf4j`, `@Builder`) to reduce boilerplate.

## Architecture Layers
- **Controller Layer**:
  - Never expose JPA entities in controllers. Use DTOs for all requests and responses.
  - All endpoints MUST return the standardized `ApiResponse<T>` wrapper.
  - Use `@Valid` for request body validation.
  - Annotate with `@Tag` and `@Operation` for OpenAPI documentation.
- **Service Layer**:
  - Encapsulate all business logic here.
  - Use `BusinessException` for error scenarios to ensure consistent error responses.
- **Persistence Layer**:
  - All entities should extend `BaseEntity` to inherit common auditing fields (`id`, `createdAt`, `updatedAt`, `createdBy`, `updatedBy`).
  - Use Liquibase for all schema changes (`src/main/resources/db/changelog`).
- **Mapping**:
  - Use **MapStruct** for all mappings between Entities and DTOs. Do not write manual mapping logic.

## Logging & Observability
- **Logging**: Use `@Slf4j`. Include correlation IDs or business keys in logs. Use appropriate levels (`INFO`, `WARN`, `ERROR`, `DEBUG`).

## Testing Standards
- **Unit Tests**: Always write JUnit 5 unit tests for business logic in the Service layer.
  - Use Mockito for dependencies.
  - Use AssertJ for fluent assertions (`assertThat`).
- **Integration Tests**: Use `@SpringBootTest` and `MockMvc` for verifying full API flows.

## API Standards
- Use kebab-case for URL paths.
- Follow RESTful naming conventions.
- Ensure all new features are reflected in the `README.md`.
